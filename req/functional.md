# 기능 요구사항 (Functional Requirements)

데이터 원본: [`discovery/test-design.json`](../discovery/test-design.json) — 캐릭터/질문/점수 로직은 이 파일을 단일 출처(SSOT)로 사용한다.

## 1. 시작 화면 (Intro)

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-1 | MUST | 테스트 제목("나의 전생 영혼동물"), 한 줄 소개, **시작하기** 버튼을 표시한다. |
| FR-2 | SHOULD | "약 1분 · 7문항" 같은 소요 시간/문항 수 안내를 표시한다. |
| FR-3 | COULD | 지금까지 참여자 수 / 인기 결과 등 가벼운 소셜 프루프를 표시한다. |

## 2. 질문 플로우 (Quiz)

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-4 | MUST | 7개 질문을 한 번에 하나씩 순서대로 보여준다. |
| FR-5 | MUST | 각 질문의 보기를 탭하면 즉시 다음 질문으로 넘어간다(별도 "다음" 버튼 불필요). |
| FR-6 | MUST | 진행률(예: 3/7 또는 진행 바)을 표시한다. |
| FR-7 | SHOULD | 이전 질문으로 돌아가 답을 수정할 수 있다(뒤로 가기). |
| FR-8 | MUST | 선택한 보기의 (축, 극, 가중치)를 클라이언트 상태에 누적한다. |
| FR-9 | SHOULD | 질문 전환에 가벼운 애니메이션/트랜지션을 준다. |

## 3. 채점 (Scoring)

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-10 | MUST | 3개 축(A/B/C) 각각 poleA·poleB 점수를 합산해 큰 쪽 극을 채택한다. |
| FR-11 | MUST | 세 축의 극 조합으로 캐릭터 코드(`A{1\|2}-B{1\|2}-C{1\|2}`)를 만들어 8유형 중 하나로 매핑한다. |
| FR-12 | MUST | 채점은 100% 클라이언트에서 수행한다(네트워크 요청 없음). |
| FR-13 | MUST | 동점이 발생하지 않아야 한다 — 설계상 보장(축별 가중치 합 홀수). 구현은 이 불변식을 깨지 않는다. |

## 4. 결과 화면 (Result)

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-14 | MUST | 결과 캐릭터의 이름·동물·이모지·태그라인·설명·강점을 카드 형태로 표시한다. |
| FR-15 | MUST | 해당 유형의 **희귀도(%)** 를 강조 표시한다(예: "전체의 6% · 최고 레어"). |
| FR-16 | MUST | 결과는 고유 URL `/r/[code]` 를 가진다(예: `/r/A2-B1-C2`). |
| FR-17 | MUST | 결과 URL 진입 시 해당 캐릭터의 OG 메타(제목/설명/이미지)가 채워져 카톡/SNS 미리보기에 카드가 뜬다. |
| FR-18 | SHOULD | "다시 하기" 버튼으로 시작 화면으로 돌아간다. |
| FR-19 | COULD | 같은 축에서 "반대 성향" 또는 "잘 맞는 영혼"(궁합) 1종을 함께 보여준다. |

## 5. 공유 (Share) — 핵심 바이럴

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-20 | MUST | 결과 화면에서 **공유하기** 동작을 제공한다(링크 복사 최소 지원). |
| FR-21 | MUST | 캐릭터별 공유 카피(`shareCopy`)를 공유 텍스트에 자동 포함한다. |
| FR-22 | SHOULD | 카카오톡 공유(링크+썸네일 카드)를 지원한다. |
| FR-23 | SHOULD | 인스타 스토리 등에 올릴 수 있는 **결과 카드 이미지**를 생성/다운로드할 수 있다. |
| FR-24 | SHOULD | OG 이미지를 캐릭터 코드별로 동적 생성한다(`/og/[code]`). |
| FR-25 | COULD | 친구의 결과 코드를 입력해 **궁합 점수**를 보는 재참여 루프를 제공한다. |

## 6. 데이터/콘텐츠

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-26 | MUST | 질문·캐릭터·점수 규칙은 코드에 하드코딩하지 않고 `discovery/test-design.json` 스키마를 따르는 데이터로 분리한다. |
| FR-27 | SHOULD | 새 테스트(다른 주제)를 추가할 수 있도록 테스트 단위로 데이터가 구조화돼야 한다. |
